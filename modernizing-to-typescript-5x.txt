Modernizing janux-people.js Build Process
Date: 2026-03-19
Updated: 2026-03-30

Overview
--------
Replace the Node 12 + Gulp 4 build pipeline with a modern Node 20 + npm scripts
setup using the TypeScript compiler directly. The goal is to eliminate unnecessary
build tool abstraction for a library that only needs to transpile TypeScript to
CommonJS with declaration files.

Current Stack
-------------
- Node 12
- TypeScript 3.1.8
- Gulp 4 (orchestration)
- gulp-typescript (TS compilation)
- gulp-tslint + TSLint (linting) [NOTE: TSLint is abandoned]
- del (directory cleaning)
- gulp-mocha + Mocha 2.x (testing)
- gulp-typedoc + TypeDoc 0.8.0 (documentation)

Target Stack
------------
- Node 20 (runtime is Node 22; engines field set to >=20)
- TypeScript 5.x
- npm scripts (orchestration)       — replaces Gulp entirely
- tsc directly (compilation)        — replaces gulp-typescript
- ESLint + @typescript-eslint (linting) — replaces abandoned TSLint
- rimraf (directory cleaning)       — replaces del/clean tasks
- Vitest + Chai (testing)           — replaces gulp-mocha + Mocha 2.x
- TypeDoc (documentation)           — replaces gulp-typedoc

Why Drop Gulp?
--------------
The current Gulp tasks are minimal: clean, lint, compile, test, watch, and docs.
None of these require a streaming build system. The TypeScript compiler handles
compilation and watch mode natively, and npm scripts can orchestrate the remaining
steps. Removing Gulp eliminates a layer of abstraction and several dependencies
(gulp, gulp-load-plugins, gulp-load-subtasks, gulp-typescript, gulp-tslint,
gulp-sourcemaps, gulp-concat, gulp-mocha, gulp-typedoc).

Why Not tsup / esbuild / Rollup / Vite?
----------------------------------------
Those tools are valuable for bundling (tree-shaking, ESM/CJS dual output, asset
handling). This project only transpiles TypeScript to CommonJS with declaration
files — tsc does this natively with zero overhead or configuration complexity.

Why Vitest Instead of Mocha + ts-node?
---------------------------------------
Node 22 ships with built-in TypeScript type-stripping. When Mocha loads a .ts
test file via import(), Node 22 intercepts the call, strips the types, and treats
the file as an ES module — bypassing ts-node and tsx entirely. Any test file that
contains a require() call then fails at runtime because require is not defined in
ES module scope. This is a fundamental incompatibility between Mocha's module-
loading strategy and Node 22's native TypeScript support.

Vitest avoids this entirely. It has native TypeScript support via its own transform
pipeline and does not depend on ts-node, tsx, or Node's module hook chain. Test
files are transformed transparently before execution, with no ESM/CJS conflicts.

Vitest is also Jest API-compatible, so test files require only minimal changes:
swap require() for import, update import paths, and add three vitest imports
(describe, it, beforeEach). All Chai assertion chains (expect(x).to.equal(y)
etc.) remain unchanged because Chai is retained as the assertion library.

Why Keep Chai?
--------------
The existing test assertions are written in Chai's BDD style (.to.equal, .to.be
.instanceof, etc.). Rewriting them to Vitest's native expect (toBe, toEqual, etc.)
would be mechanical churn with no benefit. Chai works fine alongside Vitest:
Vitest provides the test lifecycle; Chai provides the assertions.

Summary of Key Decisions
-------------------------
- lodash moved to dependencies (used in src/); follow-up: evaluate eliminating it
- generator/ excluded from published package; kept in repo for future publication
  as a separate utility package (e.g. janux-people-testdata)
- generator/ and its test files are OUT OF SCOPE for this migration; see the
  DEFERRED section below
- strict: false in tsconfig.json; follow-up: enable incrementally
- Removed obsolete deps: browserify, vinyl-source-stream, express, amdefine
- q and bluebird removed; generator/ refactoring deferred to a future project
- src/index.ts replaces hand-maintained root index.js and index.d.ts
- src/collections.ts (vendored basarat/typescript-collections) removed; see Phase 3

Phase 3 — Remove basarat/collections
------------------------------------------------------------------------------
Goal: eliminate src/collections.ts (a vendored copy of basarat/typescript-
collections) and all references to it across src/. No public API changes.

Background:
  src/collections.ts is a 2700-line vendored copy of the basarat/typescript-
  collections library included directly in the repo rather than as an npm
  package. It is used in two ways across src/impl/:
    1. collections.makeString(this) — called in toString() on every impl class;
       produces a "{prop:val,...}" string from own properties.
    2. Dictionary<string, ContactMethod> — used as the return type and internal
       implementation of the protected createContactMethodDictionary() method
       in PartyAbstract. The public methods that call it (postalAddresses(),
       phoneNumbers(), emailAddresses()) already return `any`.
  The api/ interfaces do not expose either type; the changes are entirely
  internal to impl/.

Files to update:
  src/api/Party.ts
    - Remove the three dead import lines:
        /// <reference path="../collections.ts" />
        import basarat = require('../collections');
        import collections = basarat.collections;
      (None of the types from collections are used in this file.)

  src/impl/Party.ts
    - Remove:
        /// <reference path="../collections.ts" />
        import basarat = require('../collections');
        import collections = basarat.collections;
        import Dictionary = collections.Dictionary;
    - In createContactMethodDictionary():
        Change return type:  Dictionary<string, ContactMethod>
                         →   Record<string, ContactMethod>
        Change constructor:  new Dictionary<string, ContactMethod>()
                         →   {} as Record<string, ContactMethod>
        Change setter:       contacts.setValue(contact.type, contact)
                         →   contacts[contact.type] = contact
    - In toString():
        Change: return collections.makeString(this);
             →  return JSON.stringify(this);

  src/impl/PhoneNumber.ts
  src/impl/PartyName.ts
  src/impl/StateProvince.ts
  src/impl/PostalAddress.ts
  src/impl/Country.ts
  src/impl/PersonName.ts
  src/impl/City.ts
    For each of the above:
    - Remove:
        /// <reference path="../collections.ts" />
        import basarat = require('../collections');
        import collections = basarat.collections;
    - In toString():
        Change: return collections.makeString(this);
             →  return JSON.stringify(this);

Files to delete:
  src/collections.ts

Files to update:
  test/Person.spec.ts
    - The test 'should be able to retrieve the phone numbers of a person as
      Array or Dictionary' calls phoneNumbers.getValue('work') on the result
      of person.phoneNumbers(true). getValue() is a Dictionary method; once

        phoneNumbers.getValue('work').type   →  phoneNumbers['work'].type
        phoneNumbers.getValue('work').number →  phoneNumbers['work'].number

==============================================================================
DEFERRED: Generator Refactoring
==============================================================================

The generator/ directory and its associated test files (test/UsersGenerator.spec.js,
test/UserService.spec.js) are OUT OF SCOPE for this migration. They are left
as-is (plain JavaScript) and will be addressed in a future project, most likely
when generator/ is published as a separate package (e.g. janux-people-testdata)
with its own package.json, tsconfig, and README.

Reasons for deferral:
  - generator/ uses bluebird (Promise library), lokijs, faker, and md5, all of
    which need careful handling during a TypeScript conversion
  - The generator code and tests were sufficiently complex that they bogged down
    Phase 2; isolating them avoids blocking the core library migration
  - generator/ is already excluded from the published package (not in files[])
    so deferring it has no impact on consumers

When the time comes, the generator refactoring work will include:
  - Convert generator/ from plain JS to TypeScript
  - Replace bluebird with native Promises (Promise.resolve, async/await)
  - Replace faker.random.uuid() with crypto.randomUUID() (Node 14.17+)
  - Replace lodash _.cloneDeep with structuredClone() (Node 17+)
  - Replace lodash _.isObject singleton guards with simple if (!instance) checks
  - Remove .asCallback() calls (bluebird-specific)
  - Convert test/UsersGenerator.spec.js and test/UserService.spec.js to
    TypeScript using the same Vitest + Chai pattern as the core test files
  - Add faker, lokijs, md5 back to devDependencies if still needed

FOLLOW-UP ACTIONS (post-migration, separate tickets)
==============================================================================

1. Enable strict TypeScript incrementally
   Enable individual flags one at a time in tsconfig.json, fixing errors at
   each step: noImplicitAny → strictNullChecks → strictFunctionTypes →
   strictPropertyInitialization → ... → "strict": true.

2. Evaluate eliminating lodash from src/
   Lodash is a runtime dependency. Many utilities have native equivalents in
   ES2020+ (Array.find, Object.assign, optional chaining, etc.). Removing it
   reduces the published package's dependency footprint.

3. Publish generator/ as a separate package
   Publish as e.g. janux-people-testdata. Will require its own package.json,
   tsconfig, README, and conversion from plain JS to TypeScript (see DEFERRED
   section above).
