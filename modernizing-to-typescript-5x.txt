Modernizing janux-people.js Build Process
Date: 2026-03-19
Updated: 2026-03-30

Overview
--------
Replace the Node 12 + Gulp 4 build pipeline with a modern Node 20 + npm scripts
setup using the TypeScript compiler directly. The goal is to eliminate unnecessary
build tool abstraction for a library that only needs to transpile TypeScript to
CommonJS with declaration files.

Current Stack
-------------
- Node 12
- TypeScript 3.1.8
- Gulp 4 (orchestration)
- gulp-typescript (TS compilation)
- gulp-tslint + TSLint (linting) [NOTE: TSLint is abandoned]
- del (directory cleaning)
- gulp-mocha + Mocha 2.x (testing)
- gulp-typedoc + TypeDoc 0.8.0 (documentation)

Target Stack
------------
- Node 20 (runtime is Node 22; engines field set to >=20)
- TypeScript 5.x
- npm scripts (orchestration)       — replaces Gulp entirely
- tsc directly (compilation)        — replaces gulp-typescript
- ESLint + @typescript-eslint (linting) — replaces abandoned TSLint
- rimraf (directory cleaning)       — replaces del/clean tasks
- Vitest + Chai (testing)           — replaces gulp-mocha + Mocha 2.x
- TypeDoc (documentation)           — replaces gulp-typedoc

Why Drop Gulp?
--------------
The current Gulp tasks are minimal: clean, lint, compile, test, watch, and docs.
None of these require a streaming build system. The TypeScript compiler handles
compilation and watch mode natively, and npm scripts can orchestrate the remaining
steps. Removing Gulp eliminates a layer of abstraction and several dependencies
(gulp, gulp-load-plugins, gulp-load-subtasks, gulp-typescript, gulp-tslint,
gulp-sourcemaps, gulp-concat, gulp-mocha, gulp-typedoc).

Why Not tsup / esbuild / Rollup / Vite?
----------------------------------------
Those tools are valuable for bundling (tree-shaking, ESM/CJS dual output, asset
handling). This project only transpiles TypeScript to CommonJS with declaration
files — tsc does this natively with zero overhead or configuration complexity.

Why Vitest Instead of Mocha + ts-node?
---------------------------------------
Node 22 ships with built-in TypeScript type-stripping. When Mocha loads a .ts
test file via import(), Node 22 intercepts the call, strips the types, and treats
the file as an ES module — bypassing ts-node and tsx entirely. Any test file that
contains a require() call then fails at runtime because require is not defined in
ES module scope. This is a fundamental incompatibility between Mocha's module-
loading strategy and Node 22's native TypeScript support.

Vitest avoids this entirely. It has native TypeScript support via its own transform
pipeline and does not depend on ts-node, tsx, or Node's module hook chain. Test
files are transformed transparently before execution, with no ESM/CJS conflicts.

Vitest is also Jest API-compatible, so test files require only minimal changes:
swap require() for import, update import paths, and add three vitest imports
(describe, it, beforeEach). All Chai assertion chains (expect(x).to.equal(y)
etc.) remain unchanged because Chai is retained as the assertion library.

Why Keep Chai?
--------------
The existing test assertions are written in Chai's BDD style (.to.equal, .to.be
.instanceof, etc.). Rewriting them to Vitest's native expect (toBe, toEqual, etc.)
would be mechanical churn with no benefit. Chai works fine alongside Vitest:
Vitest provides the test lifecycle; Chai provides the assertions.

Summary of Key Decisions
-------------------------
- lodash moved to dependencies (used in src/); follow-up: evaluate eliminating it
- generator/ excluded from published package; kept in repo for future publication
  as a separate utility package (e.g. janux-people-testdata)
- strict: false in tsconfig.json; follow-up: enable incrementally
- Removed obsolete deps: browserify, vinyl-source-stream, express, amdefine
- q and bluebird removed; generator/ refactored to use native Promises
- src/index.ts replaces hand-maintained root index.js and index.d.ts

Why tests import from src/ not dist/

  src/ so that no build step is required before running tests. The build step
  (tsc) remains the source of truth for the published dist/ artefact.

Files to create:

  test/UserService.spec.ts    (converted from test/UserService.spec.js)

Phase 4 — Generator Refactoring
------------------------------------------------------------------------------
Goal: generator/ uses native Promises only; npm test still passes.

Background:
  generator/ is excluded from the published package but kept in the repo for
  future publication as a separate utility package. It currently depends on
  bluebird (Promise library) and lodash. This phase removes those dependencies
  and replaces them with native Node 20+ equivalents.

Files to update:
  generator/user-dao.js
    - Remove: var Promise = require('bluebird')
    - Remove: var _ = require('lodash'), var faker = require('faker')
      (faker was only used for faker.random.uuid(); replace with
       const { randomUUID } = require('crypto'))
    - Replace _.cloneDeep(obj) with structuredClone(obj)  (Node 17+)
    - Replace _.isObject(x) singleton guards with: if (!userDAOInstance)
    - Replace all .asCallback(callback) calls: remove callback parameter
      and return native Promise directly (all callers use .then() already)
    - Replace new Promise(resolve => resolve(x)).asCallback(cb)
      with: return Promise.resolve(x)

  generator/user-service.js
    - Remove: var _ = require('lodash')
    - Replace _.isObject(userServiceInstance) with: if (!userServiceInstance)
    - Remove .asCallback(callback) calls on DAO method returns
    - Remove callback parameters from findBy()

  generator/users-generator.js
    - Remove: var faker.random.uuid() usage
    - Add: const { randomUUID } = require('crypto')
    - Replace faker.random.uuid() with randomUUID()

  package.json — no new deps; confirm these are NOT present:
    bluebird, q (should already be absent after Phase 1 cleanup)

  Expected: all test suites still pass, including UserService and

==============================================================================
FOLLOW-UP ACTIONS (post-migration, separate tickets)
==============================================================================

1. Enable strict TypeScript incrementally
   Enable individual flags one at a time in tsconfig.json, fixing errors at
   each step: noImplicitAny → strictNullChecks → strictFunctionTypes →
   strictPropertyInitialization → ... → "strict": true.

2. Evaluate eliminating lodash from src/
   Lodash is a runtime dependency. Many utilities have native equivalents in
   ES2020+ (Array.find, Object.assign, optional chaining, etc.). Removing it
   reduces the published package's dependency footprint.

3. Publish generator/ as a separate package
   Publish as e.g. janux-people-testdata. Will require its own package.json,
   tsconfig, README, and conversion from plain JS to TypeScript.
